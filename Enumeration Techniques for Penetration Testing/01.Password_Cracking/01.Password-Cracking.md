## 🔐 Password-Cracking

### Password Cracking

A **password** (PIN / passcode / secret) is a secret used for authentication. This guide explains how passwords are stored, the differences between hashing and encryption, common cracking techniques, and useful commands and tools.

---

### 🧩 Terminology

- **Plain text**: the password as entered by a user (e.g., `@rmour123`).
- **Hash (non-reversible)**: deterministic one-way transformation of input used for storage (MD5, SHA1, SHA256, etc.).
- **Encryption (reversible)**: encoding that can be reversed with a key.
- **Salt**: additional random data combined with a password to prevent lookup/rainbow-table attacks.

---

### ⚔️ Hashing vs. Encryption

**Hashing**: one-way (non-reversible). Used to validate passwords without storing plaintext.

- ⚠️ *Weak algorithms (MD5, SHA-1) are easily cracked — avoid them for new systems.*

**Encryption**: reversible if you have the key. Use when you need to recover plaintext (rare for user passwords).

**Examples:**

- **Plain:** `@rmour123`
- **Base64 (reversible):** `QHJtb3VyMTIz` (Base64 of `@rmour123`)
- **ROT13:** simple character substitution (see: https://cryptii.com/pipes/rot13-decoder)

---

### 🔢 Common Hash Types (examples)

- **MD5** — example: `974911862f34f299a010d93ada3dea` for `@rmour123`
    
    *(note: MD5 is not secure)*
    
- **SHA-1** — example: `f37b240f8e516bc5c5d32f12d2391221bcb32e8` for `@rmour123`

## 💠 **Common Hash Types (examples)**

### 🔹 **MD5**

**MD5 (Message-Digest Algorithm 5)** is a widely used cryptographic hash function that produces a fixed **128-bit (16-byte)** hash value from an input message of any length. It was designed by **Ronald Rivest** in 1991 to replace **MD4** and is specified in **RFC 1321**. MD5 works by processing the input data in **512-bit blocks** through a series of operations including **bitwise functions (AND, OR, XOR, NOT)**, **modular addition**, and **left rotations**, resulting in a unique **128-bit message digest (hash).**

MD5 hashes are typically represented as **32 hexadecimal characters.** Even a small change in the input results in a significantly different hash due to the **avalanche effect.** While MD5 was widely used for purposes like verifying data integrity and digital signatures, it has known **security vulnerabilities (collisions)** that make it unsuitable for secure cryptographic applications today. However, it remains useful for **non-cryptographic purposes** like checksums and data partitioning because of its low computational requirements.

---

### ⚙️ **Key Points**

- 🔸 **MD5 outputs a 128-bit fixed-length hash.**
- 🔸 **Input message is padded and divided into 512-bit blocks.**
- 🔸 **Uses four rounds of 16 operations involving four logical functions.**
- 🔸 **Commonly represented as 32 hex digits.**
- ⚠️ **Vulnerable to collision attacks, hence not recommended for cryptographic security.**
- 🧮 **Still used for checksums and data integrity verification.**

---

### 🧾 **Examples**

- 🧡 **MD5** — example: `974911862f344299a010d93ada3daea` for `@rmour123` *(note: MD5 is not secure)*
- 💛 **SHA-1** — example: `f37b240f8e516bc5c5d32f12d2391221b6c03b2e8` for `@rmour123`
- 💙 **NTLM** — example: `E4180D19802C1FCFD3D8D0FF20D97909` for `@rmour123`

## 💠 **Common Hash Types (examples)**

### 🔹 **SHA-1**

**SHA-1 (Secure Hash Algorithm 1)** is a cryptographic hash function designed by the **National Security Agency (NSA)** and published by **NIST** as a **U.S. Federal Information Processing Standard**. It produces a **160-bit (20-byte)** hash value typically rendered as a **40-digit hexadecimal number**.

SHA-1 works by processing data in **512-bit blocks**, using **bitwise operations**, **modular additions**, and **compression functions** to transform the input into a unique, fixed-length output. It was once widely used for **digital signatures**, **SSL/TLS certificates**, and **version control systems** such as Git.

However, due to **collision vulnerabilities** (two different inputs producing the same hash), **SHA-1 is no longer considered secure** for cryptographic purposes. It has been replaced by more secure algorithms like **SHA-256**, **SHA-384**, and **SHA-512** from the **SHA-2 family**.

---

### ⚙️ **Key Points**

- 🔸 **SHA-1 outputs a 160-bit (20-byte) hash.**
- 🔸 **Processes data in 512-bit blocks.**
- 🔸 **Commonly represented as 40 hexadecimal characters.**
- ⚠️ **Vulnerable to collision attacks (e.g., SHAttered attack by Google in 2017).**
- 🔒 **Deprecated for secure cryptographic use; use SHA-2 or SHA-3 instead.**
- 🧮 **Still found in legacy systems and applications (e.g., Git commit hashes).**

---

### 🧾 **Examples**

- 💛 **SHA-1** — example: `f37b240f8e516bc5c5d32f12d2391221b6c03b2e8` for `@rmour123`
- 💚 **SHA-256** — example: `03ac674216f3e15c761ee1a5e255f067953623c8f9664fbaecf9a7e8b8d0d6c1` for `@rmour123`

## 💠 **Common Hash Types (examples)**

### 🔹 **NTLM**

**NTLM (NT LAN Manager)** is a **Microsoft authentication protocol** used in **Windows systems** for user logins and network authentication. NTLM uses a **challenge-response** mechanism to authenticate users without transmitting their plaintext password over the network.

In NTLM, passwords are **hashed using the MD4 algorithm** after being converted to **UTF-16 Little Endian encoding.** The resulting **128-bit hash** is stored and used in authentication exchanges between the client and the server.

Although NTLM improved upon earlier LANMAN (LM) authentication by using stronger hashing and avoiding plaintext transmission, it is now considered **insecure** due to vulnerabilities to **brute-force**, **rainbow-table**, and **relay attacks**. Microsoft has replaced NTLM with **Kerberos** in modern systems, though NTLM is still supported for backward compatibility.

---

### ⚙️ **Key Points**

- 🔸 **NTLM uses the MD4 hashing algorithm on the UTF-16 encoded password.**
- 🔸 **Produces a 128-bit (16-byte) hash value.**
- 🔸 **Used primarily for Windows authentication and SMB (Server Message Block) protocol.**
- ⚠️ **Vulnerable to relay and brute-force attacks.**
- 🔒 **Superseded by Kerberos in modern Windows domains.**
- 🧩 **Still exists in legacy systems and network configurations for compatibility.**

---

### 🧾 **Examples**

- 💙 **NTLM** — example: `E4180D19802C1FCFD3D8D0FF20D97909` for `@rmour123`

## 🔐 **Password-Cracking-Techniques**

### 💠 **Password Cracking Techniques**

Offline and online password cracking are two distinct approaches hackers use to gain unauthorized access by discovering passwords.

---

### 🧠 **Offline Password Cracking**

- 🔸 Involves obtaining encrypted password hashes from a compromised system or database.
- 🔸 Attackers work on these password hashes locally, using powerful hardware like **GPUs** and specialized tools (e.g., **Hashcat**, **John the Ripper**).
- 🔸 Without interacting with the target system, attackers can try billions of password guesses rapidly without triggering security alerts or account lockouts.
- 🔸 Requires prior access to password hash files, often extracted via **credential dumping** or **data breaches**.
- 🔸 Effective against **weak hashing algorithms** or **unsalted hashes**.
- 🔸 Used by both attackers and forensic investigators to crack passwords covertly.

---

### 🌐 **Online Password Cracking**

- 🔹 Happens in real-time by attempting to authenticate directly on the live system (**website login**, **application**, **remote access**).
- 🔹 Attackers try guessing passwords by sending login requests, either manually or via **automated bots**.
- 🔹 Limited by **network speed**, **system rate-limiting**, **account lockouts**, and **monitoring**, which can detect and block frequent failed attempts.
- 🔹 Common techniques include **brute force**, **credential stuffing**, and **password spraying**.
- 🔹 Easier to detect due to authentication attempts logged by servers.
- 🔹 Often used when password hashes are not available to the attacker.

---

### 📊 **Summary of Differences**

| 🧩 **Aspect** | ⚙️ **Offline Password Cracking** | 🌍 **Online Password Cracking** |
| --- | --- | --- |
| **Access Needed** | Stolen password hash files | Live access to authentication system |
| **Speed** | Very fast with powerful hardware | Slower, network and lockout limited |
| **Detection** | Harder to detect (offline) | Easier to detect (server logs) |
| **Tools** | Hashcat, John the Ripper | Hydra, Medusa, Burp Suite Intruder |
| **Use Case** | When hashes are available | When direct login access is needed |
