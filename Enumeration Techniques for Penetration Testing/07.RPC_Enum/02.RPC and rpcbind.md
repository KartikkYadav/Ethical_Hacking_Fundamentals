# 🔹 RPC and rpcbind

**RPC (Remote Procedure Call)** and **rpcbind** are closely related technologies used in distributed computing and network communication.

They work together to allow systems to communicate and execute remote services across a network.

---

# 🟢 1️⃣ RPC (Remote Procedure Call)

RPC is a protocol that allows a program to execute a procedure (function) on a remote system as if it were running locally.

It hides the complexity of network communication from developers and applications.

---

# 🎯 Key Features of RPC

- Execute functions remotely
- Client-server communication
- Transparency between local and remote calls
- Common in distributed systems
- Used by services such as:
  - NFS
  - NIS
  - Network management systems

---

# ⚙️ Components of RPC

## 🟣 Client

The system requesting the remote procedure execution.

---

## 🟣 Server

Hosts the procedure and returns results.

---

## 🟣 Stub

Acts as an intermediary.

### Client-side stub

- Packages request data
- Sends it across the network

### Server-side stub

- Receives request
- Executes procedure
- Returns response

---

# 🔄 RPC Workflow

```text
Client
   ↓
Client Stub
   ↓
Network
   ↓
Server Stub
   ↓
Remote Procedure
   ↓
Response Returned
```

---

# 🟢 2️⃣ rpcbind

`rpcbind` is a service that maps RPC services to the ports they use.

It acts like a directory or lookup service for RPC applications.

---

# ✅ Key Features of rpcbind

- Maps RPC program numbers to ports
- Helps clients discover RPC services
- Essential for many Unix RPC services
- Listens on:

```text
TCP/UDP 111
```

---

# 🔄 Role of rpcbind

When an RPC service starts:

1. It registers itself with `rpcbind`
2. `rpcbind` stores:
   - Program number
   - Protocol
   - Port number

When a client wants the service:

1. Client contacts `rpcbind`
2. Retrieves service port
3. Connects directly to RPC service

---

# 📘 Example — NFS

## NFS Uses RPC

When NFS starts:

```text
NFS registers with rpcbind
```

Example program number:

```text
100003
```

---

# 🧪 Client Workflow

## Step 1 — Query rpcbind

Client connects to:

```text
Port 111
```

to ask:

```text
"Which port is NFS using?"
```

---

## Step 2 — rpcbind Responds

Returns:

```text
Port number for NFS service
```

---

## Step 3 — Direct Communication

Client communicates directly with NFS service.

---

# ⚖️ RPC vs rpcbind Comparison Table

| Feature | RPC | rpcbind |
|---|---|---|
| Purpose | Execute remote procedures | Map RPC services to ports |
| Role | Enables remote communication | Service discovery |
| Port Usage | Dynamic ports | TCP/UDP 111 |
| Interaction | Client ↔ Server | Client ↔ rpcbind |
| Dependency | Uses rpcbind to locate services | Required by many RPC services |

# 🛠 Useful Enumeration Commands

## View RPC Services

```bash
rpcinfo -p <target-ip>
```

---

## Nmap RPC Scan

```bash
nmap -sV -p 111 --script=rpcinfo <target-ip>
```

---

## Check rpcbind Locally

```bash
systemctl status rpcbind
```

---

## View Listening Ports

```bash
ss -lntup | grep 111
```

# ⚠️ Security Risks

RPC and rpcbind may expose:

- Internal services
- NFS shares
- System information
- Weakly secured RPC services

Misconfigurations can lead to:

- Information disclosure
- Unauthorized access
- Service abuse

---

# 🛡 Hardening Recommendations

## Disable Unused RPC Services

Reduce attack surface.

---

## Restrict Port 111

Allow only trusted hosts.

---

## Use Firewalls

Limit:

```text
111/TCP
111/UDP
```

---

## Restrict NFS Access

Use:

```text
/etc/exports
```

carefully.

---

## Enable Authentication

Avoid anonymous or weak access.
