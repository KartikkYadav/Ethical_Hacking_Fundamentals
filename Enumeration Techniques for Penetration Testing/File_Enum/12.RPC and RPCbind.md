# 🌐 RPC and RPCbind

## 🔹 RPC and RPCbind

RPC (Remote Procedure Call) and `rpcbind` are closely related but serve different purposes in distributed computing and network communication.

---

# 🟢 1️⃣ RPC (Remote Procedure Call)

RPC is a protocol that enables a program to execute a procedure (function) on a remote system as if it were local.

It abstracts the complexities of network communication, allowing developers to focus only on the logic of the procedure call.

---

## 🎯 Key Features

- **Functionality**
  - Allows execution of functions on a remote machine

- **Transparency**
  - The calling program does not need to know whether the procedure is local or remote

- **Portability**
  - Used in distributed systems such as:
    - NFS (Network File System)
    - Distributed applications
    - Network management systems

---

# ⚙️ Components of RPC

## 🟣 Client

Makes the request to execute a remote procedure.

---

## 🟣 Server

Hosts the procedure and sends the result back to the client.

---

## 🟣 Stub

Acts as an intermediary.

### Client-side stub

- Packages the request
- Sends it over the network

### Server-side stub

- Receives the request
- Invokes the actual procedure
- Sends back the response

---

# 🔄 RPC Communication Flow

```text
Client
   ↓
Client Stub
   ↓
Network
   ↓
Server Stub
   ↓
Remote Procedure
   ↓
Response Returned
```

---

# 🟢 2️⃣ RPCbind

`rpcbind` is a service that manages and maps RPC services to the ports they are listening on.

It is essential for the functioning of RPC services on Unix/Linux systems.

---

# ✅ Key Features

## 🔹 Port Mapper

Maps RPC services to their respective ports.

---

## 🔹 Service Discovery

Clients query `rpcbind` to determine which port an RPC service is using.

---

## 🔹 Runs on Port 111

By default, `rpcbind` listens on:

```text
TCP/111
UDP/111
```

---

# 🔄 Role in RPC

When an RPC service starts:

1. It registers itself with `rpcbind`
2. `rpcbind` stores:
   - Program number
   - Protocol
   - Port number

When a client wants to access the service:

1. Client contacts `rpcbind`
2. Retrieves service port information
3. Connects directly to the RPC service

---

# 📘 Example in Context — NFS

NFS (Network File System) uses RPC for its operations.

---

## 🧪 Example Workflow

### Step 1 — NFS Starts

NFS registers with `rpcbind`.

Example program number:

```text
100003
```

---

### Step 2 — Client Queries rpcbind

Client connects to:

```text
Port 111
```

and asks:

```text
"Which port is NFS using?"
```

---

### Step 3 — rpcbind Responds

Returns the NFS service port.

---

### Step 4 — Client Communicates with NFS

Client communicates directly with the NFS service.

---

# ⚖️ RPC vs RPCbind Comparison Table

| Feature | RPC | RPCbind |
|---|---|---|
| Purpose | Protocol for executing remote procedures | Maps RPC services to ports |
| Role | Enables client-server communication | Service discovery and port mapping |
| Runs on Port | Dynamic ports | TCP/UDP 111 |
| Interaction | Client ↔ Server communication | Client ↔ rpcbind lookup |
| Dependency | Requires rpcbind to locate services | Required for RPC service registration |

# 🛠 Useful Enumeration Commands

## View RPC Services

```bash
rpcinfo -p <target-ip>
```

---

## Nmap RPC Scan

```bash
nmap -sV -p 111 --script=rpcinfo <target-ip>
```

---

## Check rpcbind Status

```bash
systemctl status rpcbind
```

---

## View Listening Ports

```bash
ss -lntup | grep 111
```

# ⚠️ Security Risks

Misconfigured RPC services may expose:

- Internal services
- NFS exports
- System information
- Weakly secured RPC endpoints

Potential risks include:

- Information disclosure
- Unauthorized access
- Enumeration opportunities

---

# 🛡 Hardening Recommendations

## Disable Unused RPC Services

Reduce attack surface.

---

## Restrict Port 111

Allow access only from trusted systems.

---

## Use Firewall Rules

Restrict:

```text
111/TCP
111/UDP
```

---

## Secure NFS Exports

Configure:

```text
/etc/exports
```

carefully.

---

## Enable Authentication

Avoid anonymous or weak access.
