# 💀 Impacket-wmiexec

## 🔹 Impacket-wmiexec

`impacket-wmiexec` is a tool from the Impacket suite that enables remote command execution on Windows systems by abusing Windows Management Instrumentation (WMI) over DCOM.

It is a fileless alternative to tools like `psexec` and is commonly used for:

- Stealthy post-exploitation
- Lateral movement
- Remote administration
- Active Directory operations

Compared to `psexec`, it avoids uploading executables and creating Windows services, making it quieter in many environments.

---

# 🟢 Use Cases

- Fileless remote command execution
- Stealthier lateral movement in Windows environments
- Evasion of detections tied to:
  - Service creation
  - Scheduled tasks
  - Binary uploads

- Post-exploitation in Active Directory environments
- Remote administration using WMI/DCOM

---

# ⚙️ How It Works

1. Authenticates to the target over SMB using:
   - Passwords
   - NTLM hashes
   - Kerberos tickets

2. Uses WMI over DCOM/RPC to remotely spawn a process

3. Executes the provided command on the target system

4. Retrieves output through SMB shares such as:
   - `ADMIN$`
   - `C$`

5. Cleans up execution artifacts without dropping binaries

---

# 🔥 Key Features

- No executable upload to disk
- Fileless remote execution
- Semi-interactive command execution
- Supports:
  - Password authentication
  - NTLM hash authentication
  - Kerberos authentication

- Command output returned to attacker machine
- Similar to native PowerShell/WMI execution techniques

---

# 🧾 Authentication Format

```bash
[domain/]username[:password]@<target>
```

---

# 📌 Authentication Notes

- Use quotes for passwords containing special characters:

```bash
impacket-wmiexec domain/user:'P@ssw0rd!'@target
```

---

# 🟣 Pass-the-Hash

```bash
impacket-wmiexec -hashes LMHASH:NTHASH user@target
```

---

# 🟣 Kerberos Authentication

```bash
impacket-wmiexec -k -no-pass administrator@server.corp.local
```

---

# ⚠️ Requirements & Notes

- WMI/DCOM must be enabled on the target
- RPC ports must be accessible:
  - TCP `135`
  - Dynamic RPC ports

- Requires local administrator privileges
- May still generate:
  - WMI event logs
  - Process creation logs

- Can be blocked by:
  - Windows Firewall
  - Restrictive GPOs
  - EDR solutions

# 🔍 wmiexec vs psexec vs smbexec

| Feature | `wmiexec` | `psexec` | `smbexec` |
| --- | --- | --- | --- |
| Fileless | Yes | No | Partial |
| Uploads Binary | No | Yes | No |
| Creates Service | No | Yes | Yes |
| Uses WMI | Yes | No | No |
| Interactive Shell | Semi | Semi | Semi |
| Noise Level | Lower | High | Medium |

# 🚩 Common Flags

| Flag | Description |
| --- | --- |
| `-hashes` | Use NTLM hashes |
| `-k` | Use Kerberos authentication |
| `-no-pass` | Do not prompt for password |
| `-dc-ip` | Specify Domain Controller IP |
| `-target-ip` | Specify target IP |
| `-codec` | Set output encoding |
| `-debug` | Enable debug output |

---

# 📝 Notes on Detection

- Less noisy than:
  - `psexec`
  - `smbexec`

- May still generate:
  - WMI event logs
  - RPC/DCOM activity logs
  - Process creation events

- Often detected through:
  - WMI consumer activity
  - PowerShell telemetry
  - Sysmon process creation events
  - EDR behavioral monitoring

---

# 🛡️ Defensive Recommendations

- Restrict WMI/DCOM access
- Limit local administrator privileges
- Monitor:
  - WMI activity
  - RPC/DCOM traffic
  - Remote process creation

- Enable:
  - Sysmon
  - PowerShell Logging
  - Windows Event Forwarding

- Use:
  - MFA
  - Credential Guard
  - LAPS

# 🔹 Summary

`impacket-wmiexec` is one of the most effective Impacket tools for stealthy remote command execution through WMI/DCOM.

It provides:

- Fileless execution
- Semi-interactive shells
- Pass-the-Hash support
- Kerberos authentication support

It is heavily used during:

- Penetration Testing
- Red Team Operations
- Active Directory Assessments
- Post-Exploitation Activities

Because it avoids service creation and executable uploads, it is often quieter and more evasive than traditional SMB execution tools like `psexec`.
