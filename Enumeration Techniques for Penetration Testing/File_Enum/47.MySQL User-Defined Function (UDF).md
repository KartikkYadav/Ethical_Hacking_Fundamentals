# 📘 MySQL User-Defined Functions (UDFs)

## 📖 Overview

MySQL **User-Defined Functions (UDFs)** can be abused to achieve operating-system command execution when a MySQL server is misconfigured and an attacker possesses elevated database privileges (such as file-related privileges and write access to plugin locations).

> **Note:** This technique is typically discussed in post-exploitation and privilege-escalation contexts during **authorized security assessments**.

---

# 🟢 1. Configure MySQL for UDF Assessment (High-Level)

## 🟩 Step 1 — Locate MySQL Configuration Files

Common locations include:

```text
/etc/my.cnf
/etc/mysql/my.cnf
~/.my.cnf
```

---

## 🟩 Step 2 — Review Configuration

Common settings affecting plugin loading and file handling:

```ini
[mysqld]
secure-file-priv=...
plugin_dir=...
```

Additional configuration directories are commonly included:

```ini
!includedir /etc/mysql/conf.d/
!includedir /etc/mysql/mysql.conf.d/
```

---

## 🟩 Step 3 — Restart MySQL

Restart the MySQL service after configuration changes.

---

## 🟩 Step 4 — Verify Configuration

Useful information to review:

```sql
SELECT @@secure_file_priv;
SELECT @@plugin_dir;
SELECT @@tmpdir;
SELECT @@version;
SELECT @@version_compile_machine;
```

# 🟢 3. Remote MySQL Assessment

When assessing a remote MySQL instance (with authorization), common objectives include:

- Reviewing granted privileges
- Identifying the plugin directory
- Identifying temporary directories
- Determining platform architecture
- Understanding file restrictions

---

# 🟢 4. File Access Assessment

A common security concern is whether database privileges permit reading files from the underlying operating system.

Typical verification:

```sql
SHOW VARIABLES LIKE 'secure_file_priv';
```

Followed by reviewing the file-related privileges granted to the current database account.

---

# 🟢 5. UDF Assessment Workflow (Conceptual)

## 🟩 Step 1 — Identify the Environment

Determine:

- Plugin directory
- File restrictions
- Operating system
- CPU architecture
- MySQL version

---

## 🟩 Step 2 — Prepare a Compatible Shared Library

A UDF requires a compiled shared library compatible with:

- Target architecture (32-bit / 64-bit)
- Target operating system
- Target MySQL version

---

## 🟩 Step 3 — Convert Binary Data

The shared object is commonly converted into a transferable representation before deployment.

---

## 🟩 Step 4 — Deploy the Shared Library

The library must be placed in a location that MySQL can access and load.

---

## 🟩 Step 5 — Register the UDF

Register the shared library as a user-defined function within MySQL.

---

## 🟩 Step 6 — Execute the Function

Once successfully registered, the function becomes callable through SQL.

---

## 🟩 Step 7 — Cleanup

Authorized assessments should remove:

- Temporary files
- Registered UDFs
- Test artifacts
- Other assessment remnants where appropriate

---

# 🟢 6. Requirements

Successful UDF abuse generally requires:

- High database privileges
- Ability to write files on the server
- Access to the plugin directory
- Compatible operating system and architecture
- Misconfigured MySQL security controls

---

# 🟢 7. Defensive Recommendations

- Restrict the `FILE` privilege.
- Configure `secure_file_priv`.
- Limit write access to plugin directories.
- Run MySQL using least privilege.
- Monitor creation of new UDFs.
- Audit unusual file operations.
- Restrict remote administrative access.

---

# 🟢 8. Key Security Takeaways

- UDF abuse is generally a **post-exploitation** technique.
- Proper configuration of `secure_file_priv` significantly reduces risk.
- Restricting plugin-directory write permissions helps prevent unauthorized UDF loading.
- Regular privilege reviews and auditing can detect dangerous configurations.
- Least-privilege database accounts greatly reduce the impact of a compromise.
