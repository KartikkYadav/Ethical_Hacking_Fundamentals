# SSH Remote Port Forwarding (`-R`)

**SSH Remote Port Forwarding** allows a **remote SSH server** to open a listening port and forward incoming connections **back to a service reachable from the SSH client side**.

> A port opens on the **remote system**, but the service lives on **your side** (or your internal network).

---

## When to Use Remote Port Forwarding

- Expose **local services** to a remote system  
- Bypass **NAT / firewall restrictions**  
- **Reverse pivoting** in penetration tests  
- Allow a **compromised target** to access attacker services  

---

## Syntax


ssh -R [REMOTE_BIND_ADDR]:[REMOTE_PORT]:[DESTINATION_HOST]:[DESTINATION_PORT] user@SSH_SERVER

###Parameters

## Parameters

| Parameter | Meaning |
|----------|---------|
| `REMOTE_BIND_ADDR` | IP to bind on SSH server (`127.0.0.1` recommended) |
| `REMOTE_PORT` | Port opened on SSH server |
| `DESTINATION_HOST` | Host reachable from SSH client (attacker side) |
| `DESTINATION_PORT` | Service port on destination |
| `-N` | Tunnel only (no shell) |

### Basic Example

    ssh -R 9090:localhost:22 user@remote_host

### Explanation
- Port 9090 opens on remote_host
- Any connection to:

      remote_host:9090

- Is forwarded to:

      localhost:22   (SSH client side)

### Allow Remote Binding
(If You Control the SSH Server)

If you have root access to the SSH server and want to bind to a non-localhost IP, update the SSH daemon configuration.

### Edit SSH Configuration

    vim /etc/ssh/sshd_config

Add or update:
    
    AllowAgentForwarding yes
    AllowTcpForwarding yes
    GatewayPorts yes

### Restart SSH Service

    systemctl restart ssh

### Resulting Behavior

- SSH honors the bind address specified with -R
- Allows binding to:
 - A specific interface IP (e.g. 192.168.x.x)
- Requires the IP to exist on the SSH server

### Example

    ssh -N -R 192.168.3.253:8888:192.168.2.1:debian@192.168.2.253

### Expected listener:

    192.168.3.253:8888 LISTEN
    
## GatewayPorts Modes Explained
### Default (bind only to localhost)

    GatewayPorts no

### Bind to client-specified IP

    GatewayPorts clientspecified

### Bind to all interfaces

    GatewayPorts yes

⚠ Warning
GatewayPorts yes exposes the forwarded port to all network interfaces and is not recommended unless explicitly required.

### Lab Scenario
### Network Layout

- Attacker (Kali / SSH Client): 192.168.2.233
- Target (SSH Server): 192.168.2.253
- Attacker’s Web Server: 192.168.2.1:80

### Reverse Pivot – Exposing Internal Web Server

    ssh -N -R 0.0.0.0:8888:192.168.2.1:80 armour@192.168.2.253
or

    ssh -N -R 192.168.3.253:8888:192.168.2.1:80 debian@192.168.2.253

### Result

- Port 8888 opens on the SSH server (192.168.2.253)
### Accessing

    http://192.168.3.253:8888
    http://192.168.2.253:8888

### Reaches

    192.168.2.1:80

### Safer Binding (Recommended)

    ssh -N -R 127.0.0.1:8888:192.168.3.250:80 debian@192.168.253
    
### Exposing a Local Web Service to Remote Host

    ssh -N -R 8888:127.0.0.1:80 root@192.168.2.253

### Traffic flow:

    Port 8888 opens on 192.168.2.253
    → forwarded to localhost:80 (attacker/client side)

### HTTPS Example

    ssh -N -R 8443:127.0.0.1:443 debian@192.168.2.253

### plink (Windows) – Remote Port Forwarding

#### Syntax

    plink.exe -N -R [REMOTE_PORT]:[DEST_HOST]:[DEST_PORT] user@SSH_SERVER

### Examples

    plink.exe -N -R 80:127.0.0.1:80 root@192.168.2.253
    plink.exe -N -R 8080:192.168.200.10:8080 root@192.168.1.103

## Local vs Remote Port Forwarding

| Feature | Local (`-L`) | Remote (`-R`) |
|-------|-------------|---------------|
| Who accesses service | Attacker | Remote system |
| Direction | Forward | Reverse |
| Typical use | Internal access | Expose local service |
